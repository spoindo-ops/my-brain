시스템은 세 가지 주요 계층으로 구성되며, 기존 콘텐츠 생성 워크플로우(Writer/Designer)와 독립적으로 작동하는 API 중심의 데이터 파이프라인을 목표로 합니다.

| 계층 | 구성 요소 | 역할 및 기술 스택 제안 |
| :--- | :--- | :--- |
| **Presentation Layer** (인터페이스) | Frontend Interface (Admin Panel/Webhook Trigger) | 사용자가 체크리스트를 업로드하고 승인을 기록하는 UI 제공. |
| **Application Layer** (API/Logic) | QA Service API | 모든 데이터 검증, 상태 관리, 권한 부여 로직 처리. (Python/FastAPI 추천) |
| **Data & Persistence Layer** (저장소) | Database (PostgreSQL/MongoDB) | 시각 자산 메타데이터, QA 기록(Sign-off Log), 사용자의 승인 이력 저장. |
| **Integration Layer** (파이프라인) | Automation Script (Scheduler) | 파일 업로드 감지 및 API 호출을 자동화하는 스크립트. (Python/Airflow 또는 Cron 기반) |

### 2. 데이터 모델 설계 (Data Model)

핵심은 시각 자산과 승인 기록을 명확하게 연결하는 것입니다.

**테이블 1: `VisualAsset` (시각 자산 메타데이터)**
*   `asset_id` (PK): 고유 ID
*   `content_type`: 'YouTube Intro', 'Instagram Reel', 'Product Shot' 등
*   `asset_file_path`: 실제 파일 저장 경로 (S3 또는 로컬)
*   `visual_system`: 'Deep Navy/Gold' (시스템 준수 여부)
*   `creator_id`: 생성자 (Writer/Designer)
*   `status`: 'Draft', 'Pending Review', 'Approved', 'Rejected'

**테이블 2: `SignOffChecklist` (체크리스트 정의)**
*   `checklist_id` (PK): 고유 ID
*   `asset_type`: 어떤 자산에 대한 체크리스트인지 명시 (예: YouTube Intro Checklist)
*   `item_description`: 구체적인 검토 항목 (예: Gold 텍스트 강조 여부)
*   `required_status`: 승인 상태 정의 (Pass/Fail)

**테이블 3: `SignOffLog` (승인 기록 - 핵심 데이터)**
*   `log_id` (PK): 고유 ID
*   `asset_id` (FK): 어떤 자산에 대한 기록인지 연결
*   `checklist_id` (FK): 어떤 체크리스트를 수행했는지 연결
*   `reviewer_id`: 승인한 에이전트/사용자 ID
*   `sign_off_timestamp`: 승인 시간
*   `result`: 'Approved' 또는 'Rejected'
*   `comments`: 피드백 기록 (신뢰 자본 데이터)

### 3. API 엔드포인트 설계 (API Endpoint Design)

**A. 자산 등록 및 검토 요청:**
*   `POST /api/v1/asset/submit`: 새로운 시각 자산과 해당 체크리스트를 시스템에 등록하고 검토 대기 상태로 설정합니다.
    *   *입력:* 파일, `checklist_id`, `creator_id`
    *   *출력:* `asset_id`, `status: Pending Review`

**B. 승인 기록 및 업데이트:**
*   `POST /api/v1/signoff/record`: 사용자가 특정 자산에 대한 체크리스트를 검토하고 결과를 기록합니다. (핵심 신뢰 자본 입력)
    *   *입력:* `asset_id`, `checklist_id`, `result` ('Approved'/'Rejected'), `comments`
    *   *출력:* 성공 메시지 및 새로운 로그 ID

**C. 상태 조회:**
*   `GET /api/v1/assets/{asset_id}`: 특정 자산의 현재 QA 상태와 모든 승인 기록을 조회합니다.
*   `GET /api/v1/reports/signoff_summary?asset_id={id}`: 특정 자산에 대한 최종 승인율(신뢰 자본 지표)을 요약하여 제공합니다.

### 4. 구현 실행 계획 (Implementation Roadmap)

**Step 2: 핵심 API 구현 (2일차)**

**Step 3: 자동화 파이프라인 구축 (3일차)**

**파일명:** `qa_pipeline.py`

```python
import json
from datetime import datetime
from typing import List, Dict, Any

# --- 1. 데이터 모델 정의 (VisualAsset & SignOffChecklist 기반) ---

class VisualAsset:
    """디자인 에셋에 대한 메타데이터 모델"""
    def __init__(self, asset_id: str, file_path: str, description: str):
        self.asset_id = asset_id          # 고유 ID (UUID 권장)
        self.file_path = file_path        # 실제 파일 경로 또는 URL
        self.description = description    # 에셋 설명 (예: 'Intro_Shot_01')

class SignOffChecklist:
    """각 항목에 대한 시각적 승인 체크리스트 모델"""
    def __init__(self, asset_id: str, check_item: str, status: str, sign_off_date: str = None, comments: str = ""):
        self.asset_id = asset_id          # 연결된 VisualAsset ID
        self.check_item = check_item      # 체크 항목 (예: 'Color Palette Compliance')
        self.status = status              # 상태 ('Pending', 'Approved', 'Rejected')
        self.sign_off_date = sign_off_date if sign_off_date else datetime.now().isoformat()
        self.comments = comments          # 피드백 및 코멘트

# --- 2. 데이터 저장소 시뮬레이션 (실제로는 DB 사용) ---
# 실제 구현 시 SQLAlchemy, PostgreSQL 등을 사용하여 데이터베이스에 연결해야 합니다.
DB_ASSETS: Dict[str, VisualAsset] = {}
DB_SIGN_OFFS: Dict[str, List[SignOffChecklist]] = {}

def save_asset(asset: VisualAsset) -> str:
    """VisualAsset을 저장하고 ID를 반환합니다."""
    asset_id = asset.asset_id
    DB_ASSETS[asset_id] = asset
    if asset_id not in DB_SIGN_OFFS:
        DB_SIGN_OFFS[asset_id] = []
    return asset_id

def save_signoff(signoff: SignOffChecklist):
    """SignOffChecklist를 저장합니다."""
    if signoff.asset_id not in DB_SIGN_OFFS:
        DB_SIGN_OFFS[signoff.asset_id] = []
    DB_SIGN_OFFS[signoff.asset_id].append(signoff)

# --- 3. API 엔드포인트 (Flask 예시) ---
from flask import Flask, request, jsonify

app = Flask(__name__)

@app.route('/api/assets', methods=['POST'])
def create_asset():
    """새로운 VisualAsset을 시스템에 등록합니다."""
    data = request.get_json()
    if not data or 'file_path' not in data:
        return jsonify({"error": "Missing required fields"}), 400

    # 실제 파일 업로드 및 저장 로직은 여기에 통합되어야 합니다.
    new_asset = VisualAsset(
        asset_id=f"ASSET_{datetime.now().timestamp()}", # 임시 ID 생성
        file_path=data['file_path'],
        description=data.get('description', 'Untitled Asset')
    )
    
    asset_id = save_asset(new_asset)
    return jsonify({
        "message": "Asset successfully registered",
        "asset_id": asset_id,
        "status": "Pending Review"
    }), 201

@app.route('/api/signoff/<asset_id>', methods=['POST'])
def record_signoff(asset_id: str):
    """특정 에셋에 대한 시각적 승인 기록을 등록합니다."""
    if asset_id not in DB_ASSETS:
        return jsonify({"error": "Asset not found"}), 404

    data = request.get_json()
    if not data or 'check_item' not in data or 'status' not in data:
         return jsonify({"error": "Missing checklist data"}), 400

    # 체크리스트 데이터 저장
    new_signoff = SignOffChecklist(
        asset_id=asset_id,
        check_item=data['check_item'],
        status=data['status'],
        comments=data.get('comments', '')
    )
    
    save_signoff(new_signoff)

    return jsonify({
        "message": f"Sign-off recorded for {asset_id}",
        "result": new_signoff.status
    }), 200

if __name__ == '__main__':
    # 실제 운영 환경에서는 WSGI 서버(Gunicorn 등)를 사용해야 합니다.
    app.run(debug=True, port=5000)
```

3.  **처리 단계 (Backend Logic):** 백엔드는 제출된 데이터(`SignOffChecklist`)를 분석하여 `VisualTrustScore`를 산출합니다. 이는 체크리스트 항목들의 가중치(예: 품질 점수 > 마케팅 점수)를 적용하여 계산됩니다.
4.  **결과 단계 (Pricing Integration):** `VisualTrustScore`가 최종적으로 `TrustCapital_Model`에 반영됩니다. 이 데이터는 `/api/pricing/calculate` 요청 시 입력되어, **신뢰 자본 기반의 잠재 수익성(`ProposedPrice`)**을 계산하는 데 사용됩니다.
5.  **출력 단계 (Reporting):** 모든 기록은 `/api/trust/report/{productId}`를 통해 경영진에게 신뢰도와 품질 인증의 정량화된 보고서로 제공됩니다.

### 1. PostgreSQL 스키마 정의 (Schema Definition)

다음은 `VisualAsset`, `SignOffChecklist`를 포함하는 최소한의 테이블 스키마입니다.

```sql
-- Table for Visual Assets and Quality Sign-off Records
CREATE TABLE visual_assets (
    asset_id SERIAL PRIMARY KEY,
    asset_name VARCHAR(255) NOT NULL,
    asset_type VARCHAR(50) NOT NULL, -- e.g., 'Video', 'Image', 'Design'
    visual_data_url TEXT NOT NULL,  -- Link to actual asset storage (e.g., S3/local path)
    visual_style VARCHAR(100) NOT NULL, -- e.g., 'Deep Navy/Gold'
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);

-- Table for Quality Sign-off Checklist and Trust Capital Measurement
CREATE TABLE sign_off_checklists (
    sign_off_id SERIAL PRIMARY KEY,
    asset_id INTEGER REFERENCES visual_assets(asset_id) ON DELETE CASCADE,
    sign_off_date TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    qa_status VARCHAR(50) NOT NULL, -- e.g., 'Pending', 'Approved', 'Rejected'
    signer_role VARCHAR(100),        -- Who signed off (e.g., 'Designer', 'QA')
    signoff_notes TEXT,              -- Detailed feedback/justification
    trust_capital_score NUMERIC(5, 2) -- Quantitative measure of trust/quality
);

-- Table for Pricing Calculation and Linking Trust Capital
CREATE TABLE pricing_models (
    model_id SERIAL PRIMARY KEY,
    product_name VARCHAR(255) NOT NULL,
    base_cost NUMERIC(10, 2) NOT NULL,
    visual_asset_id INTEGER REFERENCES visual_assets(asset_id), -- Link to the quality proof asset
    calculated_price NUMERIC(10, 2),
    calculation_date TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);
```

### 2. FastAPI 백엔드 구현 (FastAPI Implementation)

다음은 위 스키마를 기반으로 API 엔드포인트를 처리하는 핵심 Python 코드입니다. 이 코드는 PostgreSQL 연결 및 데이터 모델 처리를 포함합니다.

**파일: `main.py`**

```python
import os
from fastapi import FastAPI, HTTPException, Depends
from sqlalchemy import create_engine, Column, Integer, String, Text, DateTime, Numeric, ForeignKey
from sqlalchemy.ext.declarative import declarative_base
from sqlalchemy.orm import sessionmaker, relationship
from datetime import datetime

# --- Configuration & Setup (Environment Variables) ---
# DB_URL 환경 변수에서 PostgreSQL 연결 문자열을 로드합니다.
DATABASE_URL = os.getenv("DATABASE_URL", "postgresql://user:password@host/dbname") 
# 실제 운영 환경에서는 .env 파일을 사용하여 보안을 유지해야 합니다.

app = FastAPI(title="Han Soo QA & Pricing API")

# --- SQLAlchemy Setup ---
Base = declarative_base()
engine = create_engine(DATABASE_URL)
SessionLocal = sessionmaker(autocommit=False, autoflush=False, bind=engine)

# --- Data Models (ORM Mapping) ---

class VisualAsset(Base):
    __tablename__ = "visual_assets"
    asset_id = Column(Integer, primary_key=True, index=True)
    asset_name = Column(String, index=True)
    asset_type = Column(String)
    visual_data_url = Column(Text)
    visual_style = Column(String)
    created_at = Column(DateTime, default=datetime.utcnow)
    signoffs = relationship("SignOffChecklist", back_populates="asset")

class SignOffChecklist(Base):
    __tablename__ = "sign_off_checklists"
    sign_off_id = Column(Integer, primary_key=True, index=True)
    asset_id = Column(Integer, ForeignKey("visual_assets.asset_id"))
    sign_off_date = Column(DateTime, default=datetime.utcnow)
    qa_status = Column(String)  # e.g., 'Approved', 'Rejected'
    signer_role = Column(String)
    signoff_notes = Column(Text)
    trust_capital_score = Column(Numeric(5, 2))
    asset = relationship("VisualAsset", back_populates="signoffs")

class PricingModel(Base):
    __tablename__ = "pricing_models"
    model_id = Column(Integer, primary_key=True, index=True)
    product_name = Column(String, unique=True)
    base_cost = Column(Numeric(10, 2))
    visual_asset_id = Column(Integer, ForeignKey("visual_assets.asset_id"))
    calculated_price = Column(Numeric(10, 2))
    calculation_date = Column(DateTime, default=datetime.utcnow)
    visual_asset = relationship("VisualAsset", back_populates="pricing")

# --- Database Initialization (Run once to create tables) ---
def init_db():
    Base.metadata.create_all(bind=engine)
    print("✅ PostgreSQL 스키마가 성공적으로 초기화되었습니다.")

# --- API Endpoints ---

# Dependency to get DB Session
def get_db():
    db = SessionLocal()
    try:
        yield db
    finally:
        db.close()

@app.on_event("startup")
def on_startup():
    init_db()
    print("🚀 API 서버가 준비되었습니다.")


@app.post("/api/qa/submit")
def submit_qa(asset_id: int, status: str, notes: str, score: float):
    """
    QA 기록을 시스템에 제출하고 신뢰 자본을 업데이트합니다.
    """
    db = SessionLocal()
    try:
        # 1. SignOffChecklist 기록 (신뢰 자본 측정)
        new_signoff = SignOffChecklist(
            asset_id=asset_id,
            qa_status=status,
            signer_role="System/User", # 실제 사용자에 따라 동적 설정 필요
            signoff_notes=notes,
            trust_capital_score=score
        )
        db.add(new_signoff)
        db.commit()
        db.refresh(new_signoff)
        return {"message": "QA 기록이 성공적으로 제출 및 신뢰 자본이 업데이트되었습니다.", "record_id": new_signoff.sign_off_id}
    except Exception as e:
        db.rollback()
        raise HTTPException(status_code=500, detail=f"DB 처리 중 오류 발생: {e}")
    finally:
        db.close()

@app.post("/api/pricing/calculate")
def calculate_price(product_name: str, visual_asset_id: int):
    """
    신뢰 자본을 기반으로 제품 가격을 계산합니다. (예시 로직)
    """
    db = SessionLocal()
    try:
        # 1. 필요한 데이터 조회
        asset = db.query(VisualAsset).filter(VisualAsset.asset_id == visual_asset_id).first()
        if not asset:
            raise HTTPException(status_code=404, detail="해당 시각 자산(Visual Asset)을 찾을 수 없습니다.")

        # 2. 신뢰 자본 기반 가격 계산 로직 (예시: 기본 비용 * 품질 계수)
        base_cost = 100.00 # 예시 기본 비용
        trust_score = asset.signoffs[0].trust_capital_score if asset.signoffs else 1.0 # 첫 번째 QA 기록의 점수를 신뢰 자본으로 사용

        calculated_price = base_cost * (1 + trust_score * 0.5) # 신뢰도에 따라 가격을 증액

        # 3. PricingModel 저장
        new_model = PricingModel(
            product_name=product_name,
            base_cost=base_cost,
            visual_asset_id=visual_asset_id,
            calculated_price=round(calculated_price, 2)
        )
        db.add(new_model)
        db.commit()
        return {"product": product_name, "calculated_price": round(calculated_price, 2), "trust_score_used": trust_score}
    except HTTPException as e:
        raise e
    except Exception as e:
        db.rollback()
        raise HTTPException(status_code=500, detail=f"가격 계산 중 오류 발생: {e}")
    finally:
        db.close()
```

## Trust Capital Mapping Table 및 가격 책정 변수 정의 (Developer 전달용)

이 문서는 '신뢰 자본(Trust Capital)'을 시각적 품질 인증 기록과 연결하여 프리미엄 가격 책정의 정당성을 확보하기 위한 데이터 모델 및 가격 책정 변수를 정의합니다.

### 1. Trust Capital Mapping Table 필드 구조 정의 (데이터 모델)

개발자가 API 및 데이터 흐름을 구현할 수 있도록, 신뢰 자본(Trust Capital)과 가격 결정 요소를 매핑하는 테이블의 최종 필드 구조는 다음과 같습니다.

| 필드명 (Field Name) | 데이터 타입 (Data Type) | 설명 (Description) | 연관 항목 (Linked Item) | 필수 여부 (Required) |
| :--- | :--- | :--- | :--- | :--- |
| **Product_ID** | String | 제품/라인 고유 식별자 (예: 에덴, 비바더마 등) | - | 필수 |
| **Visual_SignOff_Count** | Integer | 해당 제품 라인에 대해 'Visual Sign-off'를 받은 총 횟수. (신뢰 자본의 양적 측정치) | Designer/QA 시스템 | 필수 |

| **Trust_Capital_Score** | Float | 계산된 신뢰 자본 점수. (QA 통과율 및 Sign-off 수를 기반으로 산출) | 시스템 계산 결과 | 필수 |
| **Price_Multiplier** | Float | 최종 가격에 적용할 승수(Multiplier). (신뢰 자본 점수에 비례하여 프리미엄 반영) | - | 필수 |
| **Pricing_Justification** | Text | 가격 책정 근거 요약. (예: "QA 통과율 98% 기반, 신뢰 자본 확보") | - | 권장 |

### 2. 신뢰 기반 가격 책정 변수 정의 (Multiplier/Base Price)

가격 결정은 **`Trust_Capital_Score`**를 핵심 축으로 하며, 이는 다음 두 가지 주요 변수의 함수로 계산됩니다.

#### A. Base Price (기준 가격)
*   **정의:** 제품의 기본 원가 또는 시장 진입 시점의 기준 가격.
*   **역할:** 모든 수익 모델의 하한선 설정. (이 값은 마케팅 전략에 따라 조정될 수 있으나, 초기에는 원가 기반으로 설정합니다.)

#### B. Price Multiplier (가격 승수)
*   **정의:** **`Trust_Capital_Score`**를 바탕으로 프리미엄을 부여하는 배율.
*   **계산 로직 (초기):**
    $$ \text{Price\_Multiplier} = f(\text{Trust\_Capital\_Score}) $$
    *   **Level 1 (Standard Trust):** $\text{Trust\_Capital\_Score} < 60\% \implies \text{Multiplier} = 1.0$ (원가 경쟁 포지션)
    *   **Level 2 (Verified Trust):** $60\% \le \text{Trust\_Capital\_Score} < 85\% \implies \text{Multiplier} = 1.1 \sim 1.3$ (프리미엄 시작)
    *   **Level 3 (Premium Trust):** $85\% \le \text{Trust\_Capital\_Score} \le 100\% \implies \text{Multiplier} = 1.4 \sim 2.0$ (최고 프리미엄 포지션)

#### C. 최종 가격 계산
$$ \text{Final Price} = \text{Base\_Price} \times \text{Price\_Multiplier} $$

### 3. Developer 참고 사항 (API 및 데이터 흐름)

1.  **데이터 입력:** 모든 시각적 승인 기록(`Visual_SignOff_Count`, `QA_Pass_Rate`)은 시스템에 실시간으로 입력되어야 합니다.
2.  **핵심 연결고리:** $\text{Trust\_Capital\_Score}$는 마케팅 전략(예: 국내 시장 vs. 해외 시장)에 따라 적용되는 **`Price_Multiplier`**를 결정하는 유일한 변수입니다.
3.  **API 엔드포인트:** 기존 설계된 API 엔드포인트(`/api/pricing/calculate`)는 이 `Trust Capital Mapping Table`의 데이터(특히 $\text{Trust\_Capital\_Score}$)를 입력받아 동적으로 $\text{Price\_Multiplier}$를 반환하도록 수정되어야 합니다.

## 1. PostgreSQL 스키마 최종 필드 확정 (SQL DDL)

Business에서 정의한 Trust Capital Mapping Table 및 신뢰 기반 가격 책정 변수를 시스템적으로 구현하기 위해, 다음 세 테이블에 대한 PostgreSQL 스키마(DDL)를 확정합니다.

### 1. `pricing_models` 테이블
제품 라인별로 최종적인 가격 책정 로직과 승수(Multiplier)를 저장합니다.

```sql
CREATE TABLE pricing_models (
    model_id SERIAL PRIMARY KEY,
    product_id VARCHAR(50) UNIQUE NOT NULL, -- Product_ID 매핑
    base_price NUMERIC(10, 2) NOT NULL,     -- Base_Price (기준 가격)
    trust_capital_score NUMERIC(5, 4) NOT NULL DEFAULT 0.0, -- Trust_Capital_Score (신뢰 자본 점수)
    price_multiplier NUMERIC(3, 2) NOT NULL DEFAULT 1.0, -- Price_Multiplier (가격 승수)
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    CONSTRAINT fk_product_id FOREIGN KEY (product_id) REFERENCES visual_assets(product_id) ON DELETE CASCADE
);
```

### 2. `visual_assets` 테이블
시각적 자산(제품 라인)의 마스터 정보를 저장합니다.

```sql
CREATE TABLE visual_assets (
    product_id VARCHAR(50) PRIMARY KEY, -- Product_ID (고유 식별자)
    asset_name VARCHAR(100) NOT NULL,   -- 제품 라인 이름 (예: 에덴, 비바더마)
    visual_style VARCHAR(50) NOT NULL,  -- 적용된 시각 스타일 (예: Deep Navy/Gold)
    base_price NUMERIC(10, 2) NOT NULL,  -- Base_Price (기준 가격)
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);
```

### 3. `sign_off_checklists` 테이블
QA 및 시각적 승인 기록을 저장하여 신뢰 자본의 근거를 확보합니다.

```sql
CREATE TABLE sign_off_checklists (
    sign_off_id SERIAL PRIMARY KEY,
    product_id VARCHAR(50) NOT NULL REFERENCES visual_assets(product_id), -- 연결된 제품 라인
    visual_signoff_count INTEGER NOT NULL DEFAULT 0,                  -- Visual_SignOff_Count (승인 총 횟수)
    qa_pass_rate NUMERIC(4, 2),                                         -- QA_Pass_Rate (품질 통과율)
    signoff_details JSONB,                                              -- 상세 승인 기록 저장 (JSONB 사용으로 유연성 확보)
    status VARCHAR(50) NOT NULL DEFAULT 'Pending',                     -- 현재 상태 (Pending, Approved, Rejected)
    signoff_date TIMESTAMP WITH TIME ZONE,                            -- 승인 날짜
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);
```
