### 🛠️ 실행 계획: `pack_apply.py` 안정화 및 E2E 테스트 확보

**목표:** `pack_apply.py` 실패 원인을 근본적으로 해결하고, 전체 자동화 파이프라인(스크립트 → 디자인 적용 → 편집)의 데이터 무결성과 실행 안정성을 100% 보장한다.

#### Phase 1: 실패 원인 심층 분석 및 수정 (Root Cause Analysis & Fix)

1.  **실패 로그 검토:** 이전 `pack_apply.py` 실행 시 발생한 **모든 에러 로그(stdout/stderr)**를 확보하여 어떤 단계에서 데이터 불일치, API 오류, 파일 경로 문제 중 어느 것이 핵심 원인인지 정확히 진단한다.
2.  **데이터 흐름 재검증:** 스크립트 내의 모든 데이터 변환 및 외부 API 호출 지점(`pack_apply` 내부)을 역추적하여, 'Trust Capital' 계산 모델과 시각 자료 적용 로직 간의 매핑이 완벽하게 이루어지는지 검토한다.
3.  **안정화 수정:** 실패 원인을 기반으로 `pack_apply.py` 스크립트 내의 **오류 처리(Error Handling)** 로직을 강화하고, 특히 파일 I/O 및 외부 API 통신 부분에 대해 재시도(Retry) 메커니즘을 도입하여 안정성을 확보한다.

#### Phase 2: End-to-End 테스트 안정성 확보 (E2E Stability Assurance)

1.  **테스트 환경 격리:** 자동화 스크립트가 실제 운영 환경과 동일한 조건에서 실행될 수 있도록, **Mocking 전략**을 도입하여 외부 의존성(API 호출, 파일 시스템 접근)으로부터 핵심 로직을 분리한다.
2.  **QA 모듈 통합 강화:** Designer와 협의하여, 기존에 설계된 'Visual Validator Module'의 자동화 스크립트가 `pack_apply` 실행 결과와 **100% 일치**하는지 검증하는 테스트 케이스를 개발하고 이를 파이프라인에 통합한다.
3.  **Self-Verification Loop 구축:** 모든 단계(스크립트 실행 → 데이터 저장 → 시각 자료 생성 → 최종 QA 로그 기록)가 성공적으로 완료되었는지 자동으로 확인하는 **자기 검증 루프(Self-Verification Loop)**를 구현한다. 이 루프는 실패 발생 시 즉시 알림을 발생시키고, 자동 복구 또는 수동 개입 지점을 명확히 제시해야 한다.

#### Phase 3: 실행 및 보고 (Execution & Reporting)

1.  **수정 스크립트 적용:** 분석된 내용을 바탕으로 `pack_apply.py`를 수정하고, 모든 테스트가 통과할 때까지 반복적으로 실행하여 안정성을 확보한다.
2.  **최종 안정성 보고:** E2E 테스트 성공 여부, 평균 실행 시간, 주요 병목 지점 및 해결된 오류 사항을 요약한 최종 안정성 보고서를 작성하여 공유한다.
